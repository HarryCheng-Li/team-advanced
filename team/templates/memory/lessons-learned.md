# 经验教训库

## 成功的经验 ✅

### LLM-001: 数据库设计提前评审
- **场景**: 用户登录功能
- **做法**: database-designer 完成后，architect 评审
- **结果**: 避免了后期 Schema 变更
- **复用建议**: 所有涉及数据库的功能都应评审

### LLM-002: 并行 Bug 排查
- **场景**: 网站性能问题
- **做法**: 2 个 bug-hunter 从不同角度并行探索
- **结果**: 定位时间缩短 50%
- **复用建议**: 复杂问题始终使用并行探索

## 失败的教训 ❌

### LLM-003: 未确认需求直接开发
- **场景**: 报表功能
- **问题**: 未澄清报表格式，开发后发现不符合需求
- **损失**: 2 天返工
- **改进**: Phase 0 需求澄清必须输出验收标准

### LLM-004: 跳过测试直接交付
- **场景**: API 接口开发
- **问题**: 未写测试，生产环境发现边界情况 Bug
- **损失**: 线上故障
- **改进**: 测试覆盖率必须达到 90%

## 模式总结

```yaml
高成功率模式:
  - 需求澄清 → 验收标准确认
  - 并行探索复杂问题
  - 代码审查 → 安全审查 → 测试审查

高风险模式:
  - 跳过需求澄清
  - 单人处理复杂问题
  - 无审查直接合并
```
